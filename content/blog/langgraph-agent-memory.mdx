# Beyond Chat History: Using Rules as Agent Memory

When your agent loses track of something the user said three turns ago, the response goes sideways. The user gets a generic answer, repeats themselves, or loses trust. Recency-based memory (just keeping the last N messages) is the usual fix, but a question in turn 7 might connect back to a constraint mentioned in turn 1, and the logical relationship between those two turns has nothing to do with how similar the words are.

[InputLayer](https://github.com/inputlayer/inputlayer) gives you a way to reason about this. You store conversation turns as facts, write rules that derive connections between them, and query a knowledge graph (a structured store of facts and their connections) for what's relevant before generating each response. The rules fire automatically as new facts arrive, keeping derived context up to date without any manual bookkeeping.

## The idea: facts + rules = derived memory

[InputLayer](https://github.com/inputlayer/inputlayer) is an incremental reasoning engine. You give it facts, write rules over them, and it keeps all derived conclusions up to date as new facts arrive. No batch jobs, no re-indexing.

`InputLayerMemory` is a memory component for [LangGraph](https://github.com/langchain-ai/langgraph) agents that stores conversation turns as **facts** and uses **rules** to derive what's relevant.

The schema is minimal. Two base relations:

```
memory_turn(thread_id, turn_id, role, content, timestamp)
memory_topic(thread_id, turn_id, topic)
```

And three rules that derive context automatically. Each rule reads as "the conclusion on the left is true whenever the conditions on the right are satisfied." The `<-` arrow means "comes from," and `_` is a wildcard for values you don't need to name:

```
% Active topics: what this conversation is about
active_topic(ThreadId, Topic) <-
    memory_topic(ThreadId, _, Topic)

% Relevant turns: messages cross-referenced by topic
relevant_turn(ThreadId, TurnId, Role, Content, Topic) <-
    memory_turn(ThreadId, TurnId, Role, Content, _),
    memory_topic(ThreadId, TurnId, Topic)

% Topic threads: which topics are discussed together
topic_thread(ThreadId, TopicA, TopicB) <-
    memory_topic(ThreadId, _, TopicA),
    memory_topic(ThreadId, _, TopicB),
    TopicA != TopicB
```

So the first rule says: "a topic is active in a thread if any message in that thread mentions it." The second links messages to their topics. The third finds which topics appear together in the same thread.

That's it. These rules create a **live knowledge graph** of the conversation. When you add a new turn, InputLayer incrementally updates only the conclusions affected by the new facts. No full recomputation, no stale results.

## A conversation in action

Let's trace what happens during a real conversation. You're building an ML pipeline and asking for help:

| Turn | Role | Message |
|------|------|---------|
| 1 | user | I'm building a machine learning pipeline in Python. |
| 2 | assistant | Great! What stage? Data prep, training, or deployment? |
| 3 | user | Training. The model is slow on our GPU cluster. |
| 4 | assistant | For performance, consider mixed precision and DataLoader workers. |
| 5 | user | We also have trouble with our REST API for predictions. |

### What gets stored (base facts)

Each turn is inserted as a `memory_turn` fact. Topics are auto-extracted and stored as `memory_topic` facts:

```
memory_topic("chat-1", 1, "ml")
memory_topic("chat-1", 1, "python")
memory_topic("chat-1", 3, "ml")
memory_topic("chat-1", 3, "performance")
memory_topic("chat-1", 4, "performance")
memory_topic("chat-1", 5, "api")
```

### What gets derived (automatically)

The moment those facts are inserted, the rules fire. Here's what the KG has derived:

**Active topics.** The conversation spans 4 areas:
```
active_topic("chat-1", "api")
active_topic("chat-1", "ml")
active_topic("chat-1", "performance")
active_topic("chat-1", "python")
```

**Relevant turns.** Messages cross-referenced by topic:
```
relevant_turn("chat-1", 1, "user", "I'm building an ML...", "ml")
relevant_turn("chat-1", 1, "user", "I'm building an ML...", "python")
relevant_turn("chat-1", 3, "user", "Training is slow...",   "ml")
relevant_turn("chat-1", 3, "user", "Training is slow...",   "performance")
relevant_turn("chat-1", 4, "assistant", "Consider mixed...", "performance")
relevant_turn("chat-1", 5, "user", "Trouble with REST API", "api")
```

Notice: turn 1 appears under **both** "ml" and "python". Turn 3 appears under both "ml" and "performance". The rules handle the cross-referencing automatically.

**Topic threads.** Which topics co-occur in the same conversation:
```
topic_thread("chat-1", "ml", "python")          % ml and python both tagged on turn 1
topic_thread("chat-1", "ml", "performance")     % ml and performance both tagged on turn 3
topic_thread("chat-1", "python", "performance") % python and performance both in this thread
...
```

### The "aha" moment

Now you ask: **"What about deploying with Docker and Kubernetes?"**

Before generating a response, your agent calls `memory.recall("chat-1")`. The derived context tells it:

1. **Active topics**: ml, python, performance, api, devops (the new one)
2. **Topic connections**: devops co-occurs with every prior topic in this thread
3. **Relevant turns**: the ML training discussion (turns 1, 3, 4) and the API discussion (turn 5) are both relevant, since they share topics with the new message

The rule-derived memory returns the full context: you're deploying an ML pipeline (Python, training on GPUs) with an API layer, and now you want to containerize it. The rules connected "Docker" to the earlier ML/API discussion because devops topics co-occur with all the prior ones in this thread.

## The proof tree: explainable memory

You can ask InputLayer **why** a particular turn is relevant:

```
.why ?relevant_turn("chat-1", 3, Role, Content, "performance")
```

And it returns a proof tree:

```
relevant_turn("chat-1", 3, "user", "Training is slow...", "performance")
├── because memory_turn("chat-1", 3, "user", "Training is slow...", 3000)
└── because memory_topic("chat-1", 3, "performance")
```

This is the derivation chain, a structured explanation of **why** this memory is relevant. You can include this in your LLM's prompt so it has the reasoning chain alongside the context. Instead of guessing which earlier messages matter, it sees exactly which facts and rules produced each piece of context.

## Code: adding memory to a LangGraph agent

Here's how you wire `InputLayerMemory` into a LangGraph agent. Three nodes form the loop: recall context, generate a response, then store the new turn.

```python
from inputlayer import InputLayer
from inputlayer.integrations.langgraph import InputLayerMemory
from langgraph.graph import StateGraph, END
from typing import TypedDict, Any

class ChatState(TypedDict, total=False):
    thread_id: str
    new_message: dict
    context: dict
    response: str

async with InputLayer("ws://localhost:8080/ws", username="admin", password="...") as il:
    kg = il.knowledge_graph("my_agent")
    memory = InputLayerMemory(kg=kg)
    await memory.setup()

    graph = StateGraph(ChatState)
    graph.add_node("recall", memory.recall_node(state_key="context"))
    graph.add_node("respond", respond_fn)
    graph.add_node("store", memory.store_node(state_key="new_message"))

    graph.set_entry_point("recall")
    graph.add_edge("recall", "respond")
    graph.add_edge("respond", "store")
    graph.add_edge("store", END)
```

Each turn through the graph:
1. **recall** -> queries the KG for derived context (topics, relevant turns, connections)
2. **respond** -> your LLM uses that context to generate a response
3. **store** -> the new message becomes a fact, rules fire, context updates

The `recall_node` and `store_node` are LangGraph-compatible node functions. They read from and write to the graph state automatically.

## Extending the ontology

The base ontology (turns + topics -> derived context) is a starting point. Here are rules you could add:

**Unresolved questions.** Questions you asked that never got a follow-up:

```
% First, find turns that already have a follow-up assistant response
has_response(ThreadId, TurnId) <-
    memory_turn(ThreadId, TurnId, "user", _, _),
    memory_turn(ThreadId, NextTurn, "assistant", _, _),
    NextTurn > TurnId

% Unresolved: user turns with no follow-up response
unresolved(ThreadId, TurnId, Content) <-
    memory_turn(ThreadId, TurnId, "user", Content, _),
    !has_response(ThreadId, TurnId)
```

**Conversation phases.** Detect when the topic shifts:

```
topic_shift(ThreadId, TurnId, OldTopic, NewTopic) <-
    memory_topic(ThreadId, TurnId, NewTopic),
    memory_topic(ThreadId, PrevTurn, OldTopic),
    TurnId = PrevTurn + 1,
    OldTopic != NewTopic
```

**Sentiment tracking.** If you extract sentiment as a topic:

```
frustrated_user(ThreadId) <-
    memory_topic(ThreadId, TurnA, "frustrated"),
    memory_topic(ThreadId, TurnB, "frustrated"),
    TurnA != TurnB
```

Each new rule extends your memory's reasoning capability without touching application code. The `recall()` method automatically picks up derived facts from any rule.

## Try it yourself

```bash
pip install inputlayer-client-dev langgraph
```

```python
from inputlayer import InputLayer
from inputlayer.integrations.langgraph import InputLayerMemory

async with InputLayer("ws://localhost:8080/ws", username="admin", password="...") as il:
    kg = il.knowledge_graph("my_agent")
    memory = InputLayerMemory(kg=kg)
    await memory.setup()

    await memory.astore("thread-1", "user", "Help with Python ML")
    ctx = await memory.arecall("thread-1")
    print(ctx["topics"])  # ["ml", "python"]
```
