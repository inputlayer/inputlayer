# Access Control Should Be a Rule, Not a Filter

The standard approach to access control in AI systems is post-retrieval filtering: retrieve a broad set of results, then filter out what the user is not allowed to see.

This has three problems. It is expensive - you retrieve more than you need. It is fragile - the filter is middleware that can disagree with your permissions system. And it is not auditable - you cannot trace which rule determined that a user could not see a document.

There is a better approach: write the access policy as a rule, and evaluate it as part of the retrieval itself.

## The Problem With Post-Retrieval Filtering

Consider a standard enterprise RAG setup. An employee asks a question. Your system retrieves the top 40 documents by similarity, then applies a permissions check to filter results the user is not allowed to see.

The permissions check is typically a separate system - a call to your IAM layer, a JOIN against your access control tables, a middleware component that evaluates role-based permissions.

Two problems emerge in practice.

First, the permissions system and the retrieval system have no shared state. If a permission was revoked 20 minutes ago, your cache may still show the old state. The filter may pass documents the user is no longer authorized to see.

Second, you cannot answer: "Why was this document returned to this user?" The filter either passed or it did not. There is no derivation trace showing which role assignment, which policy, which organizational hierarchy led to the access decision. In regulated industries, that trace is not optional.

## Access Control as a Rule

Write the access policy as a logical rule and evaluate it in the same pass as retrieval.

The organizational facts:

```datalog
+employee[
    ("alice", "department", "risk_management"),
    ("bob", "department", "marketing")
]

+doc_classification[
    ("report_q3_risk", "confidential", "risk_management"),
    ("campaign_brief_07", "internal", "all")
]
```

The access rule:

```datalog
+has_access(User, Doc) <-
    employee(User, "department", _),
    doc_classification(Doc, "internal", "all")

+has_access(User, Doc) <-
    employee(User, "department", Dept),
    doc_classification(Doc, "confidential", Dept)
```

Policy-filtered vector search in a single query:

```datalog
?has_access("alice", Doc),
 document(Doc, Embedding),
 Similarity = cosine(Embedding, QueryVec),
 Similarity > 0.7
```

This returns only documents Alice has access to, ranked by relevance - in one pass. The access decision and the retrieval are part of the same computation.

## Why This Matters for Audits

Every document returned has a complete derivation trace:

- Alice has access to `report_q3_risk` because...
- `doc_classification("report_q3_risk", "confidential", "risk_management")` is true, and...
- `employee("alice", "department", "risk_management")` is true, and...
- these match the `has_access` rule for confidential documents

When an auditor asks why this employee saw this document, the answer is a complete chain linking the access decision to the organizational data that produced it.

## When Permissions Change

When Alice moves from risk management to marketing, one fact changes:

```datalog
-employee("alice", "department", "risk_management")
+employee("alice", "department", "marketing")
```

Every access derivation that depended on Alice being in risk management is automatically retracted. The change propagates without a manual cache flush or a delayed sync.

The access policy is the rule. The data is the state. The engine maintains consistency between them.
